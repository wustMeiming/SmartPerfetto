② 机制：animation 回调中同步调用 CustomScroll_longFrameLoad，执行了 59.01ms

③ 阻塞原因：CustomScroll_longFrameLoad 是应用自定义的帧内同步负载

④ 系统因素：CPU 均频 2157MHz / 峰频 3072MHz（约70%），未达峰值

📚 背景知识：渲染管线
Choreographer 的 ANIMATION 回调在主线程帧窗口最早阶段执行，完成后才进入 TRAVERSAL（measure/layout/draw）。在 ANIMATION 回调中执行耗时操作会整体推迟 syncFrameState，导致 RenderThread 迟迟无法开始 DrawFrame。当前 trace 表现：主线程 60ms 的 animation 耗时导致 RenderThread 的 DrawFrames 仅用 1.88ms（负荷轻），说明瓶颈完全在主线程 animation 回调，而非 GPU 或 SF 侧。

① 症状：主线程 Q4b=80.4%，帧超时 18.66ms（2.2×）

5. 优化建议

[CRITICAL] workload_heavy — CustomScroll_longFrameLoad 同步 animation 回调（6帧，86%）

[App 层] 最高优先级：

  1. 将耗时操作移出 Choreographer ANIMATION 回调：CustomScroll_longFrameLoad 在 animation 阶段执行了 47-59ms（模拟真实业务耗时），需重构为：

    // 错误模式（当前实现）
valueAnimator.addUpdateListener {
    // CustomScroll_longFrameLoad —— 直接调用耗时操作
}

// 正确模式：将重负载异步化
coroutineScope.launch(Dispatchers.Default) {
    val result = doHeavyWork // 移到后台线程
    withContext(Dispatchers.Main) {
        applyResultToView(result) // 仅在主线程更新 UI
    }
}

  2. 分帧执行（Chunking）：如果业务逻辑必须在主线程，使用 Handler.post 或 Choreographer.postFrameCallback 将计算分散到多帧，每帧不超过 4ms（120Hz 帧预算为 8.33ms，留余量）。

  3. 分析实际生产场景对应点：CustomScroll_longFrameLoad_1/2/3 表明有 3 种变体，需逐一排查对应的真实业务逻辑（图片解码？JSON解析？本地数据库查询？），针对性异步化。

[MEDIUM] shader_cache_miss — Vulkan Pipeline 冷启编译（1帧，14%）

[App 层]：

  • 预编译 Pipeline Cache：使用 VkPipelineCache 持久化缓存到磁盘（VK_PIPELINE_CACHE_DATA_UUID），下次冷启时从文件加载，避免运行时 JIT 编译。
  • HWUI 层：Android 12+ 通过 renderEffect 或提前渲染一帧触发 shader 热身；可在 Application.onCreate 中提前绘制一帧 offscreen 触发 pipeline 编译。
  • 预期效果：首帧 postAndWait 阻塞从 8.58ms 降至 <1ms。
